# Функциональные требования и эпики

## Эпик — Мастер предоценки (брифинг)

### Требования

- На каждой вкладке мастера предоценки — явная кнопка «Сохранить» с сохранением данных только текущей вкладки и обратной связью (успех / ошибка).
- Виджеты справочника: название, описание и превью-картинка (импорт из анкеты xlsx, хранение в `data/uploads/widgets`).
- Вкладка «ФС + очереди»: **полный каталог ФС** (все пункты из `fs_catalog`), сопоставленные с выборами решений/виджетов помечаются как matched; несопоставленные — «Нет» в очередях выделено красным; **ручной выбор** — клик по ячейке очереди на любом пункте каталога (в т.ч. несопоставленном) переключает Да/Нет и сохраняет пункт с `source=manual`. Иерархическая таблица как в Excel (группа → пункт ФС), колонки **№ (префикс группы и пункта)** / пункт ФС / **тип функционала** / виджеты / все очереди / Да·Нет по очередям (без отдельной колонки F); расшифровка и формулировки заказчика — в **карточке пункта** (модалка по клику); пункт активен, если хотя бы в одной очереди «Да»; группы и пункты отсортированы по префиксу; сворачивание по группе; итоги SP только по «Да»; drag-drop очередей.
- **НСИ → ФС** в админке: справочник с колонками №, Пункт/Расшифровка, Тип, НСИ SP, НМД НСИ; inline-редактирование; учёт использования в брифингах. При создании брифинга — снимок актуального НСИ; правки НСИ не меняют существующие оценки.
- **НСИ → ФС: управление каталогом** — создание пустых групп (маркер `item_type='group'`), копирование группы/пункта как черновик «(копия)», reorder групп и пунктов (↑↓ и drag-and-drop), перенос пункта в другую группу (drop на заголовок или «Переместить в группу»); при смене порядка — пересчёт префиксов 1, 2, 3… и 1.1, 1.2…
- **Справочник гипотез (Lean Canvas, MVP)** — вкладка «Гипотезы»: название (G), ЦА (C), уровень зрелости (B), виды деятельности — названия листов xlsx (M:N, расширяемый справочник), проблематики (E) и решения у каждой проблематики; импорт `npm run import:hypotheses --workspace=server`. Мастер предоценки — в следующих итерациях.
- Карточка пункта ФС в брифинге: расшифровка НСИ, формулировки заказчика, подпункты `custom_lines`, флаг «не актуален для заказчика» (задел под ФТТ/ТЗ).

### Критерии приёмки

- Кнопка «Сохранить» видна на всех 6 вкладках; единый стиль; после успешного сохранения показывается «Сохранено».
- В списке виджетов (мастер и админка) отображаются название, описание и миниатюра из файла; клик по миниатюре на вкладках «Решения + виджеты» и «ФС + очереди» открывает просмотр в полном размере (закрытие — ✕, Escape, клик по фону).
- На вкладке «ФС + очереди» отображается полный каталог ФС; группы и пункты отсортированы по префиксу; в строке группы виден префикс (1, 2, … 11); группы без сопоставленных пунктов свёрнуты по умолчанию; строка группы показывает сводные Да/Нет; несопоставленные пункты показывают «Нет» красным; итоговая строка суммирует SP только по «Да»; выбор пункта — кликом по ячейке очереди (Да/Нет), без колонки F; клик по названию пункта открывает карточку.
- Колонка «Тип функционала» заполнена из каталога ФС (лист «2.ФС для Заполнения», col F): Базовый, Проф, Экспертный, в Разработке и др.
- Админка «НСИ → ФС»: сохранение правок, кнопка «Где использ.» показывает брифинги; новый брифинг получает снимок НСИ на момент создания.

## Эпик — Параметры оценки (Stage 1)

### Требования

- НСИ `project_types`: Кейс, Проф-мини, ПРОФ, КОРП, БЗ (наследует Кейс); ставки и коэффициенты численности.
- Вкладка «Параметры оценки» (перед «Параметры РП»): критерии продавца, авто/ручной тип проекта, риски C51–C57, орг. объём по очередям, категория численности и коэффициенты C62–C68.
- Вкладка «Параметры РП»: ставки по очередям из НСИ с ручным override; риски — read-only со вкладки оценки.
- API: CRUD типов проекта; GET/PATCH assessment; `computeAutoProjectType`, `computeRisks`, `computeOrgVolume` на сервере.

### Критерии приёмки

- Пользователь открывает вкладку «Параметры оценки», видит критерии, авто-тип, риски, орг. объём; сохраняет.
- «Параметры РП» показывает ставки по очередям из НСИ.
- Админка: вкладка «Типы проекта» в CatalogLinks.

### Stage 2 (не в scope)

- Полный фазовый расчёт 73–90, полные формулы Excel.

## Эпик — Варианты оценки и снимки КП

### Контекст

На базе **заполненной очереди** (брифинг) нужны **произвольные варианты оценки** с тремя осями изменений и двумя режимами хранения: **живой пересчёт** и **фиксация для клиента**.

### Требования

#### База

- Брифинг хранит **эталон**: ФС + очереди, орг. объём, параметры оценки и РП, включение фаз по умолчанию.
- База не дублируется в вариантах — вариант хранит только **дельту (patch)** относительно базы.

#### Живые сценарии (`assessment_scenarios`)

- Несколько именованных сценариев внутри одного брифинга; **всегда пересчитываются** при изменении базы.
- Три типа пресетов (можно комбинировать):
  1. **Сократить фазы** — вкл/выкл этапов r73–r90 (`phase_enabled` по очередям).
  2. **Сократить ФС** — исключение пунктов каталога (`fs_excluded`); пересчёт SP (C20/C21/D20), типа проекта и зависимых фаз.
  3. **Другая технология** — тот же ФС и фазы, иная технология/ставка по очереди (`queue_technology` → C32).
- UI: вкладка **«Варианты оценки»** — список сценариев, редактор дельты, сравнение **База | Сценарий 1 | …** (суммы по фазам и итого, дельта ₽ и %).
- При изменении базы после последнего редактирования сценария — бейдж **«устарел»** (информационно; пересчёт автоматический).

#### Снимки КП (`briefing_assessment_snapshots`)

Снимок — **замороженная копия** итогов КП на момент фиксации. Live-база и live-сценарии после этого **продолжают пересчитываться**; снимок — нет.

**Кнопка «Зафиксировать КП»** доступна:
- на **базе** (без выбранного сценария) — фиксируется эталон «Параметры РП»;
- на **живом сценарии** — фиксируются дельта сценария и сравнение с базой.

Можно создать **несколько снимков** на один брифинг (в т.ч. несколько по одному сценарию).

##### Три независимые оси при создании

| Ось | Варианты | Назначение |
|-----|----------|------------|
| **Объект** | База / сценарий | Что зафиксировано как «вариант КП» |
| **Глубина** | Стандартная / расширенная | Сколько данных сохраняется в БД |
| **Метка** | «Отправлено клиенту» | Учёт версии, ушедшей наружу (на расчёт не влияет) |

##### Стандартная фиксация (по умолчанию)

Сохраняется минимум для показа и сравнения КП:

- `name`, `frozen_at`, `scenario_id`, `scenario_overrides` (копия дельты сценария или `null` для базы);
- `results` — итоги на момент клика: сравнение ОТ по фазам (база | сценарий), `grandTotal`, SP по очередям;
- `base_revision` — метка `briefing.updated_at` на момент фиксации.

**Назначение:** показать клиенту цифры, сравнить варианты, убедиться, что через время суммы в снимке не «поплыли» после правок базы.

##### Расширенная фиксация (+ галочка в модалке)

Всё из стандартной **плюс** `extended_dump`:

- полная копия `assessment` (параметры РП, фазы, риски, очереди…);
- полная копия `fs_items`;
- `accuracy_pct`, `team_fte_sum`.

**Назначение:** аудит и **юридическое воспроизведение** расчёта — «из чего сложилась сумма», даже если каталог ФС и формулы позже изменятся.

##### UI (текущее состояние)

- Список снимков: название, дата, метки **«· клиенту»** и **«· расшир.»**.
- Просмотр снимка: таблица сравнения **База | Сценарий** и строка **«Итого ОТ на момент фиксации»** — одинаково для стандартного и расширенного снимка.
- Подпись при расширенном снимке: «полный дамп сохранён».
- Live-таблица сравнения остаётся доступной при открытом снимке (можно зафиксировать следующий вариант).
- Явное предупреждение: **снимок ≠ актуальный live-расчёт** после изменения базы.

**Не реализовано** (беклог #14): отдельный **просмотр и экспорт** `extended_dump` — данные пишутся в БД, UI для аудита/выгрузки дампа — в планах.

##### Отличие от live-сценария

| | Live-сценарий | Снимок КП |
|--|---------------|-----------|
| Пересчёт при правке базы | Да | Нет |
| Редактирование | Да | Нет (только удаление) |
| Полный `assessment` в записи | В базе брифинга | Только в расширенном снимке (`extended_dump`) |

#### Расчёт

- `resolveScenarioAssessment(base, scenarioDelta) → effectiveAssessment` → существующий пайплайн `computeAllPhaseBases` / `computeAllPhaseProds`.
- При исключении ФС — предупреждения о зависимых фазах (например 5.1 от SUM C76:C77).

#### Фаза 5.1 — рабочая документация (r81)

Переключатель в **Параметрах РП** (блок «БД и документация»), три позиции:

1. **Только документация** — часы C45 × C32 × C63 (+ резерв 10% от фаз).
2. **Только видео** — часы на запись роликов × C32 × C63 (+ резерв).
3. **Документация + видео** — сумма обеих частей (+ резерв).

**«Ни одного»** — фаза r81 выключена в таблице фаз (как и другие этапы).

Параметры: `rd_delivery_mode`, `c45_rd_hours`, `rd_video_hours` в `phase_calc_params`.

#### Фазы 5.2 / 5.3 — формат обучения (r84, r85)

В таблице **«Обучение»** (строки 47–49) — колонка **«Вид обучения»** в каждой строке:

| Вид | Видимые колонки строки | Расчёт |
|-----|------------------------|--------|
| **Группы** | все (как сейчас): чел./группа, часов на группу, E, F, G, H | текущие формулы F×D, командировки |
| **Вебинары** | **E**, **кол-во вебинаров**, **резерв** (на разбор вопросов) | `N × 4 × C32 × (1 + резерв)` |

**По строкам (отдельно):**

- **Строка 47** → вклад в фазу **5.3** (r85, пользователи).
- **Строка 48** → вклад в фазу **5.3** (r85, исполнители).
- **Строка 49** → фаза **5.2** (r84, админы); в групповом режиме + C46 (подготовка к обучению).

**Авто по умолчанию** (до ручной правки РП), по значению **E** в строке:

- строка 47: группы, если E ≤ 100; иначе вебинары;
- строка 48: группы, если E ≤ 300; иначе вебинары;
- строка 49: группы, если E ≤ 100; иначе вебинары.

**Параметры** (`phase_calc_params`, по очереди):

- `training_row_format` — `groups` | `webinar` на строку 47/48/49 (+ флаг ручной правки);
- `webinar_count` по строке (дефолт **10**, редактируемо);
- `webinar_qa_reserve` **по строке** — колонка в таблице (дефолт **0,2** = 20%); в режиме «Группы» колонка скрыта.

**Итог фазы** при смешанном формате (напр. 47 — группы, 48 — вебинары): сумма вкладов строк; командировки только по строкам в режиме «Группы».


- Создание сценария из пресета «только фазы»; сравнение с базой в одной таблице.
- Сценарий «−ФС» пересчитывает SP и итоги; база не меняется.
- Сценарий «другая технология» меняет ставку очереди без изменения состава ФС.
- Стандартный снимок сохраняет суммы; просмотр через месяц не меняется при правке базы.
- Расширенный снимок сохраняет полный дамп в БД; просмотр итогов — как у стандартного.
- Несколько снимков на брифинг; расширенный снимок помечается «· расшир.».
- Изменение базы пересчитывает live-сценарии и не трогает снимки.
- *(план)* Просмотр и экспорт `extended_dump` для аудита (беклог #14).

### Этапы реализации

1. Модель сценария, `resolveScenarioAssessment`, UI сравнения, пресет **фазы**. ✅
2. Пресет **−ФС**, предупреждения зависимостей. ✅
3. Пресет **технология**. ✅
4. Снимки: стандартная фиксация, список, просмотр итогов, несколько снимков. ✅ (тестирование)
5. Расширенная фиксация (запись дампа в БД). ✅ (тестирование)
6. Просмотр и экспорт `extended_dump`; бейдж «устарел» у live-сценария. 🔴
