# Функциональные требования и эпики

## Эпик — Мастер предоценки (брифинг)

### Требования

- На каждой вкладке мастера предоценки — явная кнопка «Сохранить» с сохранением данных только текущей вкладки и обратной связью (успех / ошибка).
- Виджеты справочника: название, описание и превью-картинка (импорт из анкеты xlsx, хранение в `data/uploads/widgets`).
- Вкладка «ФС + очереди»: **полный каталог ФС** (все пункты из `fs_catalog`), сопоставленные с выборами решений/виджетов помечаются как matched; несопоставленные — «Нет» в очередях выделено красным; **ручной выбор** — клик по ячейке очереди на любом пункте каталога (в т.ч. несопоставленном) переключает Да/Нет и сохраняет пункт с `source=manual`. Иерархическая таблица как в Excel (группа → пункт ФС → расшифровки → выбранные виджеты), колонки **№ (префикс группы и пункта)** / пункт ФС / **тип функционала** / виджеты / все очереди / Да·Нет по очередям (без отдельной колонки F); пункт активен, если хотя бы в одной очереди «Да»; группы и пункты отсортированы по префиксу (натуральная сортировка); сворачивание **по группе** (группы с сопоставлениями развёрнуты по умолчанию) и **по пункту ФС** (свёрнут по умолчанию; при развороте — расшифровки и виджеты); на **строке группы** — префикс из Excel и сводные Да/Нет по дочерним пунктам (Да, если хотя бы один дочерний пункт имеет Да в очереди); строка **итогов SP** внизу — только ячейки со статусом «Да»; виджеты — отдельные строки нижнего уровня с названием, описанием и превью; перетаскивание маркера «Да» между колонками очередей (одна активная очередь на пункт ФС; клик — переключение Да/Нет).

### Критерии приёмки

- Кнопка «Сохранить» видна на всех 6 вкладках; единый стиль; после успешного сохранения показывается «Сохранено».
- В списке виджетов (мастер и админка) отображаются название, описание и миниатюра из файла; клик по миниатюре на вкладках «Решения + виджеты» и «ФС + очереди» открывает просмотр в полном размере (закрытие — ✕, Escape, клик по фону).
- На вкладке «ФС + очереди» отображается полный каталог ФС; группы и пункты отсортированы по префиксу; в строке группы виден префикс (1, 2, … 11); группы без сопоставленных пунктов свёрнуты по умолчанию; пункты ФС свёрнуты по умолчанию (расшифровки/виджеты — при развороте); строка группы показывает сводные Да/Нет; несопоставленные пункты показывают «Нет» красным; итоговая строка суммирует SP только по «Да»; выбор пункта — кликом по ячейке очереди (Да/Нет), без колонки F; несопоставленный пункт можно включить вручную кликом по очереди.
- Колонка «Тип функционала» заполнена из каталога ФС (лист «2.ФС для Заполнения», col F): Базовый, Проф, Экспертный, в Разработке и др.; у строк виджетов наследуется от родительского пункта ФС.

## Эпик — Параметры оценки (Stage 1)

### Требования

- НСИ `project_types`: Кейс, Проф-мини, ПРОФ, КОРП, БЗ (наследует Кейс); ставки и коэффициенты численности.
- Вкладка «Параметры оценки» (перед «Параметры РП»): критерии продавца, авто/ручной тип проекта, риски C51–C57, орг. объём по очередям, категория численности и коэффициенты C62–C68.
- Вкладка «Параметры РП»: ставки по очередям из НСИ с ручным override; риски — read-only со вкладки оценки.
- API: CRUD типов проекта; GET/PATCH assessment; `computeAutoProjectType`, `computeRisks`, `computeOrgVolume` на сервере.

### Критерии приёмки

- Пользователь открывает вкладку «Параметры оценки», видит критерии, авто-тип, риски, орг. объём; сохраняет.
- «Параметры РП» показывает ставки по очередям из НСИ.
- Админка: вкладка «Типы проекта» в CatalogLinks.

### Stage 2 (не в scope)

- Полный фазовый расчёт 73–90, полные формулы Excel.

## Эпик — Варианты оценки и снимки КП

### Контекст

На базе **заполненной очереди** (брифинг) нужны **произвольные варианты оценки** с тремя осями изменений и двумя режимами хранения: **живой пересчёт** и **фиксация для клиента**.

### Требования

#### База

- Брифинг хранит **эталон**: ФС + очереди, орг. объём, параметры оценки и РП, включение фаз по умолчанию.
- База не дублируется в вариантах — вариант хранит только **дельту (patch)** относительно базы.

#### Живые сценарии (`assessment_scenarios`)

- Несколько именованных сценариев внутри одного брифинга; **всегда пересчитываются** при изменении базы.
- Три типа пресетов (можно комбинировать):
  1. **Сократить фазы** — вкл/выкл этапов r73–r90 (`phase_enabled` по очередям).
  2. **Сократить ФС** — исключение пунктов каталога (`fs_excluded`); пересчёт SP (C20/C21/D20), типа проекта и зависимых фаз.
  3. **Другая технология** — тот же ФС и фазы, иная технология/ставка по очереди (`queue_technology` → C32).
- UI: вкладка **«Варианты оценки»** — список сценариев, редактор дельты, сравнение **База | Сценарий 1 | …** (суммы по фазам и итого, дельта ₽ и %).
- При изменении базы после последнего редактирования сценария — бейдж **«устарел»** (информационно; пересчёт автоматический).

#### Снимки КП (`briefing_scenario_snapshots`)

- Кнопка **«Зафиксировать КП»** на базе или на живом сценарии; снимок **не пересчитывается** при дальнейших правках базы.
- Два уровня фиксации (выбор при создании):
  - **Стандартная** (по умолчанию): overrides сценария + **итоговые результаты** (суммы по фазам/очередям, ОТ/ДО, тип проекта, SP).
  - **Расширенная**: дополнительно полный дамп `assessment` + `fs_items` + параметров на момент фиксации (аудит / юридическое воспроизведение).
- Список снимков: дата, название, сумма, признак «отправлено клиенту»; просмотр и экспорт без live-пересчёта.
- Явное предупреждение: снимок ≠ актуальный live-расчёт после изменения базы.

#### Расчёт

- `resolveScenarioAssessment(base, scenarioDelta) → effectiveAssessment` → существующий пайплайн `computeAllPhaseBases` / `computeAllPhaseProds`.
- При исключении ФС — предупреждения о зависимых фазах (например 5.1 от SUM C76:C77).

### Критерии приёмки

- Создание сценария из пресета «только фазы»; сравнение с базой в одной таблице.
- Сценарий «−ФС» пересчитывает SP и итоги; база не меняется.
- Сценарий «другая технология» меняет ставку очереди без изменения состава ФС.
- Стандартный снимок сохраняет суммы; просмотр через месяц не меняется при правке базы.
- Расширенный снимок содержит полный дамп для воспроизведения расчёта.
- Изменение базы пересчитывает live-сценарии и не трогает снимки.

### Этапы реализации

1. Модель сценария, `resolveScenarioAssessment`, UI сравнения, пресет **фазы**.
2. Пресет **−ФС**, предупреждения зависимостей.
3. Пресет **технология**.
4. Снимки: стандартная фиксация, список, просмотр.
5. Расширенная фиксация; несколько сценариев; статус «устарел».
