# Методика предварительной оценки (инвентаризация Excel vs FS_PC)

Источник: `Копия ФС Lite и ПК для предварительной оценки заказного проекта.xlsx`  
Дата инвентаризации: 2026-06-26

---

## 1. Архитектура Excel-методики

### Ключевые листы

| Лист | Назначение |
|------|------------|
| `1.Содержание проекта` | Анкета заказчика: документация, риски договора, критерии отнесения к ПРОФ/КОРП |
| `2.ФС для Заполнения` | Каталог функционала, Story Points, очереди, орг. объём (жёлтые ячейки) |
| `параметры расчета` | Справочник ставок, «попугай», коэффициенты по численности |
| `Очередь 1 Кейс-Совм` | Расчёт 1-й очереди: **Совместный запуск** + **Кейс** (две колонки оценки) |
| `Очередь 1 ПРОФ_КОРП` | Расчёт 1-й очереди: **ПРОФ/КОРП** (колонки ОТ / ДО) |
| `Очередь 2/3 ПРОФ_КОРП`, `Развитие ПРОФ_КОРП` | Аналогичные расчёты для очередей 2, 3, Развитие |
| `Оценка свод КейсПроф + очереди` | Свод по фазам × технологиям × очередям |
| `Списки` | Справочники типов проекта (K4–K7), очередей |
| `Прафс-лист на СОФТ` | Цены лицензий (строки 27–68 свода) |

### Поток данных (упрощённо)

```
1.Содержание проекта ──► критерии типа проекта (ПРОФ/КОРП)
         │
2.ФС для Заполнения ──► SP по очередям (C20/C21), орг. объём (AD5–AD7)
         │
параметры расчета ──► ставки, коэффициенты численности
         │
Очередь N [Кейс-Совм | ПРОФ_КОРП] ──► фазы 73–90, итоги 91–92
         │
Оценка свод ──► итоговое КП по технологиям
```

**Важно:** строки **27–68** на листах очередей — это **блок параметров расчёта РП**. На листе `2.ФС для Заполнения` те же номера строк — это **пункты каталога ФС**, а не параметры.

## 2. Логика технологий: Кейс / ПРОФ / КОРП

### 2.1. Типы проекта (лист `Списки`, колонка K)

| Код | Название | Когда применяется |
|-----|----------|-------------------|
| K4 | Кейс-проект | Базовый сценарий, мало критериев ПРОФ/КОРП |
| K5 | Проф/мини | Промежуточный |
| K6 | ПРОФ-проект | БП, >200 пользователей, нестандартные документы, нагрузочное |
| K7 | КОРП-проект | Технология заказчика, оптимизация БП, методология, >1000 сотр., >500 польз., ИБ |

### 2.2. Автоопределение типа проекта (C33 на листах очередей)

```
IF( E24>0 ИЛИ E2="КОРП Заказчик" ИЛИ C4>500 ИЛИ C11="ДА" ИЛИ C14="ДА"
    ИЛИ D20>0 ИЛИ C13="ДА" ИЛИ C16="ДА" ) → КОРП-проект (K7)
ELSE IF( D24>0 ИЛИ C10="ДА" ИЛИ C12="ДА" ИЛИ C4>200 ИЛИ C5>20 ИЛИ C15="ДА" ) → ПРОФ-проект (K6)
ELSE IF( C24>0 ) → Проф/мини (K5)
ELSE → Кейс-проект (K4)
```

Где:
- **C4** — число пользователей (из ФС)
- **C5** — РП/РПО/рук. отделов
- **C10–C16** — флаги из «Содержания проекта» (документы, БП, методология, нагрузочное, ИБ)
- **C24/E24/D24** — SP по типам функций (Кейс/Проф/ПРОФ/КОРП) из ФС
- **D20** — SP с требованием НМД
- **E2** — «КОРП Заказчик» если численность = 1001+

### 2.3. Ставка по типу проекта (C32)

```
C32 = IF(C33=КОРП → 6500, IF(C33=ПРОФ → 5500, иначе 5000))
```

Значения из `параметры расчета`: C3=5000 (Кейс), C4=5500 (ПРОФ), C5=6500 (Заказные/КОРП).

### 2.4. Три сценария в своде (`Оценка свод КейсПроф + очереди`)

| Колонка | Технология | Источник |
|---------|------------|----------|
| F (Совм.запуск) | Совместный запуск | `Очередь 1 Кейс-Совм`!F |
| H (Кейс) | Кейс-проект | `Очередь 1 Кейс-Совм`!Q |
| J (ПРОФ/КОРП) | ПРОФ или КОРП | `Очередь 1 ПРОФ_КОРП`!P |
| L/M/N | Очереди 2/3/Развитие | соответствующие листы |

На листе **Кейс-Совм** каждая фаза считается **дважды**: колонки F–N (Совм.запуск) и P–Y (Кейс), с разными флагами включения D/O.

На листе **ПРОФ_КОРП** — колонки **ОТ** (L/V) и **ДО** (с учётом точности C58): `V = L × (1 + C58)`.

### 2.5. Коэффициенты KORP по численности (`параметры расчета` R245–249)

Тип заказчика (C62) = категория из орг. объёма ФС (AC2: «ДО 200», «201–500», «501–1000», «1001+»).

| Параметр | до 200 | 201–500 | 501–1000 | 1001+ |
|----------|--------|---------|----------|-------|
| Коэфф. реализации функционала (C63) | 1 | 1.5 | 2 | 3 |
| Коэфф. интеграций и подготовки БД (C64) | 1 | 1 | 1.5 | 2 |
| ОПЭ ч/польз. РП/РПО/рук. (C67) | 6 | 6 | 8 | 8 |
| ОПЭ ч/польз. исполнитель (C68) | 1 | 1 | 1 | 1 |

Корректировка только РП.

## 4. Функциональный объём (лист `2.ФС для Заполнения`)

### 4.1. Story Points по пункту

- **M** — вес функционала (SP)
- **N** = SUM(O, R, U, X) — итого SP по очередям
- **O/R/U/X** — SP в очередях 1–4: `IF(очередь="ДА", M, "")`
- **Q/T/W/Z** — сценарии нагрузочного: `IF(очередь="ДА", ROUNDUP(SP/5, 0), 0)`

Включение в очередь (AC/AF/AI/AL): мега-IF по типу проекта в AC8 и флагам H–L (PM ПРОФ, PM КОРП…).

### 4.2. Тип проекта по функции (колонка G)

Значения: `Кейс-проект`, `Проф/мини`, `ПРОФ-проект`, `КОРП-проект` — влияют на C24–E24 (раскладка SP по технологиям).

## 5. Фазы проекта (строки 73–90 листов очередей)

Общая структура строки фазы:

| Колонка | Смысл |
|---------|-------|
| C | Базовая оценка (руб, до резервов) |
| D | Этап вкл (Да/Нет) |
| L / V | Производственная оценка ОТ / ДО |
| F / P | Оценка «От»/«До» без НДС (с резервами) |
| E / O | С НДС |
| M / W | Трудоёмкость для заказчика (ч) |
| N / X | Длительность (нед): `ROUNDUP((L/C32)/40×1.2 / Y, 0)` |
| Y / Z | Сумма FTE команды по ролям (Z–AE) |
| G–K / Q–U | Разбивка: продажи, РК, фонд, риски, РПО |

**Роли (FTE на фазу):** РП (Z), ФА (AA), ТА (AB), Аналитик (AC), Консультант (AD), Разработчик (AE).

**Длительность проекта (мес):** N72/4 или X72/4 (строка 91).

### Фаза 0.1. Продажа (Предпроект) — R73

- **Входы:** обычно 0 в шаблоне
- **Формула C:** 0
- **Отличия:** одинаково для всех технологий

### Фаза 0.2. Методическая проработка — R74

- **C (ПРОФ):** `D20 × C35 × C32 × C63 + IF(E7>0, C50×E7, 0)`
- **D:** C13/C14 (методология из Содержания)
- **Кейс:** D75=C14; при НЕТ → 0 в Совм.запуске, может быть в Кейсе через O75
- **Зависимости:** SP с НМД (D20), метод. часы, ставка, коэфф., командировки РГ

### Фаза 1–2. Инициация + Формирование требований — R75

- **C (ПРОФ):** `(C20+C21) × C37 × C32 × C63 + IF(E7>0, E7×C50)`
- **C (Кейс):** `(C20+C21) × C37 × C32 × C63 + IF(E7>0, E7×C50)` (меньше C37)
- **D:** всегда ДА
- **Зависимости:** SP функционала+интеграций, сбор требований, ставка, C63, регион

### Фаза 3. Проектирование ИС — R76

- **C:** `C20 × C38 × C32 × C63`
- **Кейс:** D77=НЕТ для Совм.запуска, ДА для Кейса (O77) — **ключевое отличие**
- **Зависимости:** SP функционала, проектирование/ФЕ

### Фаза 4.1. Разработка ИС — R77

- **C (ПРОФ):** `C20 × C39 × C32 × C63`
- **C (Кейс):** C39=0 → **фаза = 0** для Кейса
- **D:** ДА только для ПРОФ/КОРП

### Фаза 4.2. Интеграции — R78

- **C:** `C21 × C36 × C32 × C64`
- **Зависимости:** SP интеграций, базовый объём ПП, C64

### Фаза 4.3. Нагрузочное тестирование — R79

- **C:** `E20 × 250000` (250 тыс руб за сценарий)
- **D:** C15 (флаг нагрузочного из Содержания)

### Фаза 5.1. Подготовка к ОЭ: Настройка БД — R80

- **C (ПРОФ):** `SUM(C75:C78)×0.1 + C40×C32`
- **C (Кейс):** `C40 × C32 × C64` (без 10% от фаз 1–4)
- **D:** ДА

### Фаза 5.1. Рабочая документация — R81

- **C (ПРОФ):** `SUM(C76:C77)×0.1 + C45×C32×C63`
- **C (Кейс):** `MAX(C45×C32×C63, SUM(C77:C78)×0.1)`; D82=НЕТ для Совм., O82=ДА для Кейса

### Фаза 5.1. Миграция данных — R82

- **C:** `SUM(C76:C77) × 0.1` (ПРОФ) / `SUM(C77:C78)×0.1` (Кейс)
- **Кейс:** обычно выключена (D83=НЕТ)

### Фаза 5.1. Проектный аутсорсинг — R83

- **C:** `192000 + 96000×5 = 672000` (фикс)
- **Только ПРОФ_КОРП** (нет на Кейс-Совм в том же виде)

### Фаза 5.2. Обучение администраторов — R84/R85

- **C:** `(C46 + F49×D49) × C32 + H49×C50`
- **Зависимости:** C46, группы F49, дни H49, командировка

### Фаза 5.3. Обучение конечных пользователей — R85/R86

- **C:** `(F47×D47 + F48×D48) × C32 + SUM(H47:H48)×C50`
- **Зависимости:** орг. объём C5–C7, размер групп

### Фаза 6. Ввод в ОЭ — R86/R87

- **C (ПРОФ):** `((C5×C67 + C6×C68)×0.5 + C65) × C32 + E5×C50`
- **C (Кейс):** `(C65×C32 + E7×C50)` — упрощённо, без поминутного расчёта по ролям
- **C65:** 140 ч (ПРОФ) vs фикс. сумма (Кейс)

### Фаза 7. Поддержка в ходе ОЭ — R87/R88

- **C (ПРОФ):** `((C5×C67+C6×C68)×0.5 + C66) × C32 + E5×C50`
- **C (Кейс):** `C66 × C32`; D88=НЕТ (не в Кейс по умолчанию)

### Фаза 8.1. Документация ИБ — R88/R89

- **C:** 5 000 000 руб (фикс)
- **D:** C16 (требования ИБ)

### Фаза 8. Завершение / передача на сервис — R89/R90

- **C:** = C81 (стоимость РД)
- **D:** обычно НЕТ

### Сопровождение БД 1 год — R90/R91

- **C:** фикс (ПРОФ: 2 641 188; Кейс: 1 386 000)
- **D:** ДА
- Входит в итог, но не в трудоёмкость фаз 73–89 (отдельная строка)

## 7. Gap-анализ: Excel vs приложение FS_PC

### 7.1. Что реализовано в приложении

| Компонент | Файл | Статус |
|-----------|------|--------|
| Каталог ФС + SP + очереди | `briefingData.ts` → `fs_catalog` | ✅ Импорт из ФС |
| Автовыбор ФС из решений/виджетов/отрасли | `briefingCalc.ts` `deriveFsFromSelections` | ✅ |
| Параметры РП: ставка, точность, SP cost, фазы, команда | `briefing_params`, UI вкладка «Параметры РП» | ✅ Частично |
| Расчёт по очередям | `calculateBriefing` | ✅ Упрощённый |
| Сценарий заказчика | `briefings.scenario` | ✅ Поле есть (Кейс/ПРОФ/Совм.запуск) |
| Численность | `briefings.headcount` | ✅ Поле есть, **не в расчёте** |
| Отраслевые блоки ФС | `fs_industry_blocks` | ✅ |
| Константы (ставка, часов/день) | `constants` | ✅ Минимальный набор |
| Детальный калькулятор проекта | `client/utils/calc.ts`, `server/calc.ts` | ✅ Другая модель (этапы проекта) |

### 7.2. Формула расчёта в приложении (`briefingCalc.ts`)

```
budget_queue = SP_queue × sp_cost_rub × accuracy_mult
hours = budget / hourly_rate
duration_days = ceil(hours / (Σ team_FTE × hours_per_day))
```

Где `accuracy_mult`: low=0.85, medium=1.0, high=1.2.

**Не используется:** тип проекта, коэффициенты численности, фазовая модель, резервы РПО/фонд/НДС, командировки, нормативы на ФЕ, различие Кейс/ПРОФ/КОРП.

### 7.3. Критические пробелы

| # | Область Excel | В приложении |
|---|---------------|--------------|
| 1 | 17+ фаз с индивидуальными формулами | Нет — только агрегат по SP×очередь |
| 2 | Автоопределение Кейс/ПРОФ/КОРП (C33) | `scenario` вручную, не влияет на расчёт |
| 3 | Коэффициенты по численности (C63–C68) | `headcount` не используется |
| 4 | Три ставки (5000/5500/6500) | Одна `hourly_rate` |
| 5 | Резервы: РПО 20%, фонд, РК, риски договора | Нет в briefingCalc |
| 6 | НДС 7% | Есть в `calc.ts` проекта, не в briefing |
| 7 | ОТ vs ДО (точность C58) | `accuracy` как ±15/20%, другая семантика |
| 8 | Орг. объём (пользователи, РП, РГ, регион) | Нет |
| 9 | Нагрузочное (E20×250k) | Нет |
| 10 | Командировки (C50, H47–H49) | Нет |
| 11 | Совм.запуск vs Кейс (двойной расчёт) | Только выбор scenario, без двойной колонки |
| 12 | Очереди 2, 3, Развитие (отдельные листы) | Очереди 1–4 в ФС есть, фазовый расчёт — нет |
| 13 | Свод `Оценка свод КейсПроф` | Нет |
| 14 | Сопровождение БД, аутсорсинг, ИБ 5 млн | Нет |
| 15 | FTE по ролям на каждую фазу (Z–AE) | Одна команда на все очереди |
| 16 | Критерии из `1.Содержание проекта` | Частично (проблематики), не риски договора |
| 17 | Прайс СОФТ | Нет |
| 18 | «Попугай» C241 для нормирования | Нет |

### 7.4. Что совпадает по смыслу

- Story Points как мера функционального объёма
- 4 очереди внедрения
- Ручное включение/перенос пунктов ФС между очередями
- Редактируемая ставка и «точность» РП
- Состав команды для длительности (но формула длительности упрощена)
- Импорт фаз из сводного листа (как справочник имён, без формул)

### 7.5. Рекомендуемый порядок доработки (для будущей реализации)

1. Связать `scenario` + критерии → тип проекта → ставка (C32)
2. Подключить `headcount`/орг. объём → коэффициенты C63–C68
3. Перенести фазовые формулы (хотя бы 1-й очереди) в `calculateBriefing`
4. Добавить резервы и НДС в цепочку бюджета
5. Разделить расчёт Кейс / Совм.запуск / ПРОФ_КОРП
